the problem

AWS Batch is conservative by design

When AWS Batch provisions a server for a job, it reserves capacity for the job's worst-case resource use — peak memory AND peak CPU — simultaneously, even if those peaks never actually overlap in time.

the opportunity

Kubernetes can be smarter about this

Kubernetes separates what a job normally uses (its soft limit) from what it might briefly spike to (its hard limit). Scheduling is based on the soft limit — only 8% of peak RAM.

AWS Batch logic

reserves 64 GB RAM + 16 vCPU per job

128 GB node → 2 jobs max. Fully "full" even when 96 GB sits idle during the compute phase.

K8S / OKD logic

reserves 5.1 GB (soft limit) per job

128 GB node → ~13 jobs scheduled. One spike at a time is absorbed from reserved headroom.

The brief (<1 min) RAM spikes are unlikely to overlap. When they do, the affected job restarts automatically — at most 3 times. Zero restarts occurred in our reference run.

methodology

How we tested this — without building anything

We built a discrete-event simulation: a faithful model of job arrivals, resource use, and scheduling decisions over time. The same stream of jobs was run through both schedulers.

Stage 1 — workload generator

Creates a reproducible stream of 242 jobs drawn from realistic distributions, each with exactly the four-phase resource profile our container follows.

Stage 2 — scheduler simulation

Runs that identical job stream through the Batch rules, then the K8S rules. Tracks every EC2 dollar spent and every second a job waited in queue.

What the scheduler knows

Each job declares its resource needs at arrival — nothing more. No advance knowledge of future jobs, just like the real system.

What we measure

EC2 compute cost · queue wait time · job completion rate · SLA compliance · idle server time

Speaker note: The simulation is open-source Python, reproducible with a single command. The event list is a fixed file — both schedulers see exactly the same jobs in the same order.

The 390-second mean wait reflects a 300-second "panic threshold" — the point at which the scheduler is required to guarantee a slot. Lowering this reduces wait times but increases cost. This tradeoff is the subject of the next slide.

Speaker note: Both schedulers show identical wait times because the panic threshold — not the packing strategy — is the main driver of queue time. K8S gets its advantage from using fewer servers, not from scheduling faster.
result · tradeoff

Cost vs. responsiveness — a tunable dial

By adjusting how long the scheduler waits before reserving a server, we control the cost/speed tradeoff. K8S stays cheaper than Batch across the entire range of configurations tested.

K8S: $397.95, Batch: $402.61 across thresholds of 60, 180, 300, 600, 900, 1800, 3600 seconds. K8S is cheaper at every point.

At very high wait tolerance, cost may paradoxically worsen — long-running jobs keep servers alive longer, blocking scale-down. The simulation detects this "meta-effect" automatically.

Speaker note: Each point is one tested configuration. Lower-left is ideal (cheaper and faster). K8S dominates at every threshold tested.
reliability

Memory spikes are handled, not just avoided

When two jobs on the same K8S server simultaneously spike to peak RAM and exceed physical capacity, one job is automatically restarted — no data is lost, just time.

1. Spike collision detected

Combined RAM exceeds the node's physical capacity. One job is randomly selected as the "victim" and interrupted.

2. Job requeued automatically

The interrupted job goes back into the queue and reruns from the beginning. Up to 3 retries are allowed.

3. No data lost

This is a restart, not a permanent failure. The job completes correctly — just slightly later.

Why collisions are rare: The RAM spike phase lasts under one minute. For two jobs to collide, they would both need to enter that phase within the same ~60-second window — unlikely across 10+ co-scheduled jobs with staggered arrivals.

Speaker note: The K8S headroom strategy is sized to accommodate exactly one spike at a time. Two simultaneous spikes of the largest job type would exceed headroom — but the probability of that timing is low.
recommendation

Proceed to a prototype

The simulation provides sufficient evidence that K8S/OKD is cost-favorable without degrading service quality. The next step is a small-scale real-world validation.

What the simulation shows

K8S is cheaper at every configuration tested. Service quality is equivalent. Crash-and-retry provides a safety net. Savings are expected to grow with workload density.

What a prototype validates

Real scheduling behavior under production load. Actual crash frequency. Operational overhead of running an OKD cluster. Behavior at higher job density.

Key constraint respected

Container image is used as-is throughout. No code changes were required for the simulation and none would be required for a prototype.

Risk level: low

OKD is free and open-source. A prototype cluster can be created, tested, and torn down without long-term commitment or licensing cost.

The ask is approval to build a small prototype — not a migration decision. We are buying information, not changing the system.

Speaker note: A prototype would run a representative subset of real jobs through a real OKD cluster alongside Batch, and compare actual costs and failure rates.
appendix · glossary

Terms used in this presentation

AWS Batch
Amazon's managed service for running batch compute jobs. Provisions EC2 instances automatically based on declared job requirements.
OKD / K8S
OKD is the free, open-source upstream of Red Hat OpenShift. It runs Kubernetes (K8S) under the hood — no licensing fee.
Soft limit
How much RAM Kubernetes uses when deciding whether a server has room for another job (8% of peak RAM in our model).
Hard limit
The actual maximum RAM the job can consume at its peak. May exceed the soft limit briefly during Phase 2.
Centroid
A representative job profile. Real jobs vary around this profile following a statistical (Pareto) distribution.
Panic threshold
The maximum time a job may wait in queue before the scheduler is required to launch a dedicated server for it.
Pareto frontier
The set of configurations where no other option is strictly better on both cost AND speed simultaneously.
Discrete-event simulation
A model where time advances triggered by events (job arrivals, completions, server launches) rather than in fixed time steps.
Crash-and-replay
When a memory spike collision is detected, one affected job is restarted automatically — up to 3 times before being flagged.
EC2
Amazon's virtual server product. Cost is measured in server-hours times the hourly rate for the instance type provisioned.
